Headline findings

  • N2 (row 2), re-investigated this pass: append-only is code-confirmed MATCH; hash-chain tamper-evidence is ASPIRATIONAL because its SEC-0011 intent is unrealised end-to-end — an orphaned port with retired ADRs, not a staged rollout. pardosa-file's writer/append surface has no update/delete method and EventStore's three-method trait cannot express one either — SEC-0008:R1 holds structurally. On the hash-chain half, the prior pass's "CHE-0060 staged rollout stub pending PGN-0005 / removed when PAR-0021 lands" rationale is retracted as factually wrong: (1) the pardosa substrate already ships the precursor-hash chain — PGN-0005 is Accepted and implemented (integrity.rs:85 verify_precursor_chains, checked.rs:148, pardosa-wire/precursor.rs) with passing tamper-detection tests, not "pending"; (2) the cherry-pit capability port HashChainedEventStore (store.rs:302, exported lib.rs:75) is orphaned — zero impls, CHE-0060 Deprecated (stale 2026-05-19, host crate removed), PAR-0021 also Deprecated and retired without a PGN edge, and the PardosaEventStore the doc-comment names was retired whole (CHE-0073:R6 / CHE-0074:R8); (3) gh-report's live boot open (lifecycle.rs:863 open_with_backend → rehydrate_unchecked) does not reject discontinuity before replay, so SEC-0011:R3 is unmet at boot — precursor verification happens only on cursor reads (cursor.rs:265 stream_checked). Recorded as ASPIRATIONAL per the mission's anti-fabrication constraint: the direction is abandoned/superseded, not pending, so neither a plain MATCH nor DRIFT. A separate SEC-0011:R3 boot-load conformance row is a candidate follow-up.
  • cherry-pit-merger is the sharpest divergence (row 11): two ADRs (CHE-0069, CHE-0085) describe a designed command-side serialiser role, but no in-scope crate wires it in. Whether this is dead/staged code or wired via a mechanism outside Cargo.toml (dynamic dispatch, feature flag, future integration) cannot be resolved from the dependency graph alone.
  • Deployment topology (row 12) is code-observable — in both the gh-report tree and the companion ops repo. The gh-report tree carries a committed multi-stage crates/gh-report/Dockerfile, and gh-report-ops/cloudrun/ghreport/ holds applied Terraform: a single-instance Cloud Run service (minScale=1/maxScale=1) behind an IAP-gated load balancer, with GH_REPORT_PARDOSA_BACKEND=nats wired to the MAP JetStream cluster. The design doc's intended topology and production-backend stance both hold. The standing lesson: an infra claim is not "aspirational" until both the crate's own container manifest and the sibling <repo>-ops tree have been checked.
  • DESIGN.md's own W1/W2 corrigenda (row 13) are the strongest evidence the prose sources are self-aware of drift — the document names its own gap (ApplicationServices/publish_or_trace/InProcessEventBus never shipped) before any code comparison occurs. The C4 module inventory for gh-report's app/ confirms the admission: none of those names appear.
  • "cherry-pit-app depends on every other crate" (row 16) is narrower in practice — cherry-pit-merger and cherry-pit-storage sit outside that edge set. Combined with row 11, this suggests cherry-pit-merger's integration story is the single largest unresolved architectural question surfaced by this review.
  • Framed against the Solon thesis: the repo's bet is that a small set of enabling constraints, enforced where the type system and build graph can enforce them, collapses the space of possible mistakes. The 20 collapsed MATCH rows are exactly that promise redeemed — compile-time/ dependency-graph/CI-gate facts that agree with the documented invariant, no runtime discipline required for most of them. Rows 11, 13, and 16 are counter-evidence: places where intent outran shipped code, or wiring diverged from the stated rule. N2's hash-chain half is a different shape from those — a capability whose SEC-0011 intent is unrealised end-to-end: the substrate ships the chain, but the cherry-pit port that would expose it is orphaned (zero impls, CHE-0060/PAR-0021 both Deprecated) and the live boot open verifies nothing (SEC-0011:R3 unmet) — abandoned/superseded direction rather than unplanned drift, which is why it reads ASPIRATIONAL rather than DRIFT. None of this contradicts the thesis — a subtractive architecture still needs its unshipped or partially-wired components named honestly, which DESIGN.md's own W1/W2 notes already do for row 13, and which the corrected N2 entry now does for the hash-chain capability.
  • Net verdict distribution (24 claims evaluated, 25 verdict tags — row 2 splits into two half-verdicts): 21 MATCH, 3 DRIFT, 1 ASPIRATIONAL — the review corpus is majority-confirming, with the DRIFT/gap rows concentrated in exactly the areas the source beads already flagged as gaps (cherry-pit-merger wiring, DESIGN.md's self-admitted W1/W2 corrigenda, cherry-pit-app's narrower-than-stated edge set). The 20 MATCH claims have been pruned from the table (pruning is presentational only — every claim was still individually evaluated; full evidence lives in git history and mission beads). Two prior non-MATCH rows resolved on investigation in an earlier pass: deployment (row 12) to MATCH once the companion ops repo was scoped in, and #[non_exhaustive] (row 9) to MATCH once the gateway was confirmed to define no error enum of its own. This pass corrected one prior finding rather than fabricating a MATCH it could not derive: N2's hash-chain half (row 2) moves from a mistaken DRIFT (a grep that missed the substrate chain) to ASPIRATIONAL — but on re-investigation the ASPIRATIONAL rationale itself was corrected: the gap is an orphaned capability port with retired governing ADRs (CHE-0060/PAR-0021 both Deprecated) plus boot-load verification that SEC-0011:R3 requires but the live open path does not perform — not a "staged rollout pending PGN-0005", which is Accepted and already implemented in the substrate.
