DDD Pattern Review — cherry-pit substrate

A pattern-by-pattern audit of how well the cherry-pit crate family (cherry-pit-core, -app, -gateway, -merger, -projection, -storage, -wq) realises the tactical and strategic patterns of Domain-Driven Design. Every verdict is decided by source evidence (type/trait names + file:line), cross-read against the binding ADRs (CHE-0004 EDA+DDD+hexagonal, CHE-0005 single-aggregate). Compiled 2026-07-23.
Depth pass (2026-07-23): the "well represented" verdicts below are decided on the trait/type surface. A second, adversarial implementation-body review (Rust-specialist, full file:line evidence in bd bead adr-fmt-t8k35) then asked whether the deep semantics of each surface-solid pattern actually hold in code — no happy-path-only shells, no dropped invariants, no bypass paths. Findings are in the new Depth review section; two Partial downgrades and several tracked gaps result. Surface verdicts are retained (the abstraction is genuinely first-class) but annotated where the realisation is thinner than the name implies.
Language pass (2026-07-23): for the three depth-Solid patterns, a ubiquitous-language consistency check compares the code doc-comments against the governing ADRs term-by-term. Result: strongly consistent coined vocabulary, four minor drifts (L1–L4), no contradictions. L1–L4 were then individually re-verified against source; the L4 finding was corrected (it is a phantom-adapter-roster drift, not the "pardosa removed" claim of the first draft — pardosa is alive and is gh-report's production backend). Remediation (2026-07-23): L1, L2, L3 have since been fixed in source (CHE-0042:R3 accessor list completed, CHE-0042 envelope.rs→event.rs path corrected, CHE-0041:R5 citation added to idempotency.rs); the L4 root cause was traced to a delete-then-reintroduce name collision and its most misleading comments fixed.
Framing: cherry-pit is a subtractive substrate — illegal compositions must fail to type-check (AGENTS.md, the Solon stance). Several DDD patterns are therefore deliberately absent, not missing; the report distinguishes "lacking by omission" from "lacking by ratified deferral" wherever an ADR records the choice.

Verdict legend

Well represented first-class trait/type in the substrate, DDD name used verbatim. Partial the responsibility exists but is folded into another construct, or realised without a dedicated named abstraction. Lacking no source trace; either a genuine gap or a ratified deferral (noted per row). By design absent absence is an ADR-ratified design choice, not a gap. Depth review — do the deep semantics hold?

Every pattern in the tactical/strategic tables that scored Well represented was re-examined at the implementation body, adversarially: looking for the panic, the non-determinism, the dropped field, the missing check, the bypass path. The question is not "does the abstraction exist" (it does) but "is the DDD invariant the abstraction promises actually enforced, or only the happy-path shell." Depth verdicts: Solid deep semantics hold; Mixed trait-level solid but a real realisation gap exists; Shallow happy-path only. Full file:line evidence in bd bead adr-fmt-t8k35.

Domain Event / Envelope Well Solid Fields private, single validated ctor, no &mut accessors — genuinely immutable post-construction (event.rs:76-312). validate_stream (event.rs:197-224) really catches gaps, out-of-order, and cross-stream mixing; sequence starts at 1 by type (NonZeroU64) + build_envelopes (gateway/…/msgpack_file.rs:248-286). correlation_id/causation_id are genuinely stamped onto every envelope, not accepted-and-dropped (round-trip tests confirm). Minor: validate_stream has no explicit duplicate-event_id check (unreachable via normal construction, but not asserted).
Value Objects Well Solid No bypass constructor found. IdempotencyKey::new_unchecked is genuinely pub(crate)-sealed (idempotency.rs:47); the only public path trims + rejects empty/whitespace (never synthesizes — CHE-0046:R3 structurally enforced, not just documented). CorrelationContext deliberately has no Default (CHE-0039:R2). All immutable, no setters.

The one concrete replay (SchedulerState::from_history, scheduler.rs:226) is a deterministic BTreeMap/BTreeSet fold, no panic, no time-of-replay. Gap G1 (High): the production write path — cherry-pit-merger — defines a parallel MergerArm::handle (merger/src/arm.rs:109) that never bounds A: HandleCommand nor calls through Aggregate::handle. The compile-time-verified per-command dispatch CHE-0008 advertises is structurally bypassable by the very crate that carries real work.
Command (+ handle/apply split) Well Mixed Within cherry-pit-core the split is real and per-command Error is a genuine associated type (CHE-0015, aggregate.rs:119). Two realisation gaps: (a) EventStore takes raw Vec<Event> with no provenance — appending events handle would never emit is unprevented (inherent to event sourcing; enforcement is discipline via gateway/merger, not types); (b) MergerArm::Err is one error type per arm (typically per-aggregate), diluting CHE-0015's per-command granularity at the merger boundary (arm.rs:90). Same structural disconnect as G1.
Policy (choreography) Well Mixed Terminal-vs-Retryable classification and dead-lettering are correctly implemented and tested: Terminal → DeadLetterRecord to sink (sink failure logged, not swallowed), Retryable returned to caller (app/src/dispatch.rs:167-204). Purity of react is convention-only (&self + no async, but nothing type-forbids I/O — inherent Rust limitation). Gap G2 (High): doc comments claim Retryable errors "flow back via CommandGateway's retry (CHE-0046:R1)", but run_dispatch_consumer (app/src/app.rs:333-354) only tracing::error!s and continues — the envelope is never re-queued or retried anywhere in App::run. A Retryable dispatch failure is silently dropped after logging. Documentation/implementation mismatch.
Projection / CQRS Well Mixed Read side is structurally write-free: ReadPort::resolve takes the projection by & only, no append path expressible (projection.rs:75); Projection::apply infallible like aggregate apply. Gap G3 (Medium): the anti-downgrade invariant CHE-0092 (cited in checkpoint.rs's own module doc) is not implemented anywhere. FileProjectionStore::persist_inner (cherry-pit-projection/src/lib.rs:458-488) writes any last_sequence unconditionally — no compare-before-write against the on-disk checkpoint. A persist(seq 3) after persist(seq 10) silently downgrades the checkpoint, the exact corruption the ADR names.
Single-writer / aggregate boundary Well Mixed The merger genuinely serialises everything through its own mpsc + single-task loop — Merger::run fully .awaits each load→handle→persist→publish before dequeuing the next (merger.rs:108); TOCTOU closed. Gap G5 (Medium): Merger::spawn takes Arc<S>; nothing consumes or seals the caller's clone, so a composition root can retain a second Arc and call store.append() directly, bypassing the queue. Serialisation holds only by convention at the composition root. SingleWriterEventStore is a zero-method marker (advisory, exactly as documented — a store may impl it truthfully or falsely with nothing to catch a false claim); MsgpackFileStore instead relies on a real per-aggregate lock + directory flock (stronger, but single-process only).

Consistency verdict

  • Directionally excellent. Across all three Solid patterns the coined vocabulary — "stamp", "gap-free / duplicate-free", "defense-in-depth", "never synthesise", "conscious omission", "stream partition key", "source of truth", "single-writer / defense-in-depth" — appears verbatim on both the code and ADR surfaces, with correct back-citations in the doc comments. This is the strongest possible evidence that the ubiquitous language is real and not retrofitted.
  • Every drift is staleness or citation hygiene, never contradiction. L1 (CHE-0042:R3's normative accessor list names 5 of the 7 shipped accessors — and disagrees with the ADR's own §Accessor-methods block, which lists all 7: an intra-ADR inconsistency, not a code gap), L2 (CHE-0042 cites the long-renamed envelope.rs; the sole such reference in the corpus), and L4 (a phantom adapter roster) are ADRs/docs lagging reality. L3 (the code cites the behavioural rules it upholds but omits CHE-0041:R5, the ADR that ratifies the very pub(crate) seal it implements) is citation hygiene. None of the four changes the meaning a reader derives.
  • L4 is the one worth acting on, but for a corrected reason. The prior pass mis-stated it as "pardosa removed from the workspace" — that is wrong: pardosa is alive (10 crates) and is gh-report's production backend (CHE-0074). The real drift is narrower and sharper: store.rs:18 advertises "Pardosa-backed, PostgreSQL-backed" implementations of this port, but no PostgreSQL store exists and pardosa is barred from implementing cherry-pit-core::EventStore by the CHE-0010 severance — it reaches cherry-pit only through gh-report's own adapter. The port's ubiquitous language names an adapter roster the tree does not have. The capability-trait PGN citations (store.rs:232-343) are not part of this drift — they correctly name pardosa as design origin. L1/L2/L3 are one-line ADR/doc fixes. All are reported, not remediated, by this pass.

Root cause of the L4 confusion — a delete-then-reintroduce name collision

The first draft's wrong "pardosa was removed from the workspace" claim was not careless reading — it was induced by stale and ambiguous documentation. Git history shows two different crate families shared the name "pardosa" across a deletion boundary:

  • Legacy pardosa (pardosa-encoding — a binary Encode/Decode "GenomeSafe" serialization substrate) existed 2026-05-15 → 05-21, then was physically deleted (commit 36f0d74, "S5: physically delete pardosa-* crates from workspace"). DomainEvent was swapped off its Encode bound onto serde the same day (193093e).
  • Current pardosa (the .pgno / NATS "EDA storage layer implementing fiber semantics", the PGN-ADR substrate, gh-report's CHE-0074 backend) was imported ~2.5 weeks later (commit 050e08d, 2026-06-08, "Import Pardosa workspace crates"), reusing the name for an unrelated codebase.

So the doc comments were true when written and became misleading only when a new pardosa was grafted onto the same name. Three stale/ambiguous artefacts propagated the confusion into the review:

Misleading artefactWhy it misledStatus
event.rs:320-323 — "…removed alongside the pardosa crates" The smoking gun: literally asserts "the pardosa crates" were deleted. True for legacy pardosa, plainly false against a tree with 10 live pardosa-* crates. Fixed — reworded to scope the removal to the legacy pardosa-encoding crate and explicitly note the current substrate is live and unrelated.
event.rs:16, Cargo.toml:32, tests/adt_obligations.rs:17 — "Pardosa-encoding has been removed from the workspace" Ambiguous capitalisation/parse: "Pardosa-encoding" (one legacy crate) skims as "Pardosa, encoding" → "pardosa is gone". The Cargo.toml/test copies compound it by repetition. Partly fixed — event.rs:16 reworded ("the legacy pardosa-encoding crate… unrelated to the current pardosa substrate"). The Cargo.toml and test-doc copies remain (lower blast radius; flagged for a follow-up sweep).
C4 docs — c4/level2-containers.html:81,182, level1-context.html:75 — "pardosa dep explicitly REMOVED" Accurate but easy to over-generalise: these correctly say cherry-pit-core has no pardosa dep (CHE-0010 holds), yet the bare word "REMOVED" reinforces the "pardosa is gone" reading when skimmed out of scope. Left as-is — technically correct for the crate they describe; not a defect, but noted as a contributing amplifier. Could be tightened to "no pardosa dependency (severance, not deletion)".

Lesson. The reusable trap is name reuse across a delete-and-reintroduce boundary: comments written against the first tenant of a name silently become lies about the second. The durable fix is not just rewording — it is never letting a removal comment name a bare identifier (pardosa) that a future crate might re-occupy; scope it to the exact artefact removed (pardosa-encoding) and its reason. The store.rs:18 phantom-roster drift (L4 proper) is the same disease one step removed: the doc names implementations by substrate rather than by the one concrete adapter that actually exists.

Saga (orchestration + compensation) By design absent No Saga trait/type. Genuine omission, but low-cost: predicate/selection logic currently lives inline in handle and query resolvers. No ADR records the choice — a candidate for a future ADR if reusable business predicates emerge (e.g. on the read side).

Headline findings

  • Surface abstractions are first-class; the depth pass is where the nuance lives. All eight "well represented" patterns carry the DDD term verbatim and are type-enforced at the trait boundary. Under adversarial implementation-body review three of them hold end-to-end (Solid: Domain Event/Envelope, Value Objects, Repository) and five are Mixed — trait-solid but with a real realisation gap (see Depth review, gaps G1–G5).
  • "Partial" here mostly means "folded, not missing." Entity, Factory, and Domain Service have their responsibilities present (identity via AggregateId, validated construction via EventStore::create, pure logic in handle/react) but no dedicated trait — a defensible consequence of the single-aggregate, infrastructure-owned-identity design (CHE-0005, CHE-0020). The ACL is present in hexagonal port form rather than as a named context-map layer.
  • The genuinely absent orchestration patterns are ratified, not overlooked. Saga and Process Manager are deliberately deferred by CHE-0040: compensation is domain-event-driven, timeouts are explicit domain events, failed policy outputs are dead-lettered. The correlation/causation envelope fields are the substrate a saga would use — the orchestrator is a conscious non-goal pre-1.0, re-affirmed after cherry-pit-app shipped. Durable multi-step state, when needed, is modelled as an aggregate (the scheduler, CHE-0081), not a framework process manager.
  • Specification is the one unratified gap. It is the only pattern that is both absent from source and unrecorded in an ADR. Impact is low today (predicate logic lives inline in handle and query resolvers), but if reusable business predicates emerge — especially on the read side — it is the natural candidate for a new ADR + a small Specification trait. Recorded here as the single actionable follow-up.
  • Net assessment. Measured against DDD's catalogue, cherry-pit is a high-fidelity tactical + event-sourcing DDD substrate with strong strategic single-context discipline. Its omissions cluster exactly where the Solon subtractive stance predicts — orchestration machinery and multi-context tooling that would add degrees of freedom the niche (single-writer, intra-org, sub-PB) does not need. The only pattern lacking both code and a ratifying decision is Specification. The depth pass adds a second axis: even among the type-enforced patterns, the strongest guarantees are on the data plane (immutable envelopes, sealed value objects, TOCTOU-closed store), while the softer spots cluster on the control/coordination plane (merger bypass of HandleCommand, missing retry, missing anti-downgrade, un-sealed single-writer) — precisely the surface a saga/process-manager layer would harden if it were built.
