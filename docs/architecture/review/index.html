SEC-0011 (hash-chain tamper-evidence over persisted envelope metadata and payload bytes, R3: load rejects hash-chain discontinuity as StoreError::CorruptData). Prior-pass rationale retracted: the earlier "CHE-0060 rollout stub pending PGN-0005, removed when PAR-0021 lands" framing is factually wrong on ADR status — see the code-evidence cell. Orphaned port, retired ADRs, not a staged rollout. (1) The substrate already ships the chain. PGN-0005 is Accepted and implemented in pardosa: crates/pardosa/src/dragline/integrity.rs:85 verify_precursor_chains(), crates/pardosa/src/persist/checked.rs:148 (forged precursor_hash → typed error), and crates/pardosa-wire/src/precursor.rs precursor_hash_of — with passing tamper-detection tests. It is not "pending in pardosa source." (2) The cherry-pit capability port is orphaned. The public trait HashChainedEventStore: EventStore (crates/cherry-pit-core/src/store.rs:298, frontier_hash store.rs:304, verify_chain store.rs:317, exported at crates/cherry-pit-core/src/lib.rs:75) has zero impls in the tree. Its governing CHE-0060 is Deprecated (stale 2026-05-19, host crate removed) and the named successor PAR-0021 is also Deprecated ("retired without a PGN edge" — it will never land); the PardosaEventStore byte-adapter the doc-comment names was retired whole (CHE-0073:R6 / CHE-0074:R8). The trait's doc-comment (the prior pass's source) is itself stale. (3) Boot load does not verify. gh-report's live open path (state.rs:2089 → lifecycle.rs:863 open_with_backend → rehydrate_unchecked) folds the frontier but does not reject discontinuity before replay — so SEC-0011:R3 is unmet at boot; precursor hashes are verified only on cursor reads (cursor.rs:265 stream_checked). Per PGN-0005:R4/R6 the frontier is mutation-detection, not authentication, with no anchor seam under v0.x. DRIFT — the SEC-0011 intent is unrealised end-to-end (orphaned/retired port + unwired boot-load verification): documented but the code diverges. 2026-07-19 re-investigation (two corrections): (i) remedy mis-framed. SEC-0011:R3 mandates verification on-read, not a boot-load whole-stream verify — the earlier "no boot-load verification" framing overreaches R3 and would risk a PGN-0005:R6 anchor-seam smell. The true defect is narrower: gh-report never routes reads through the checked seam, and in fact no precursor-hash-verifying open/rehydrate constructor is wired into gh-report's boot path (stream_checked, persist/checked.rs:208, is the sole precursor-hash site; it is reachable via open_validated, lifecycle.rs:1601 → rehydrate_validated → stream_validated (validated.rs:79) → stream_checked, so open_validated genuinely runs the chain check — but gh-report's live open never calls it, routing through open_with_backend/rehydrate_unchecked instead). (ii) cross-backend divergence (new, load-bearing). The precursor-hash chain is implemented for the .pgno arm only. The NATS/JetStream rehydrate arm (rehydrate_event_frames, lifecycle.rs:608) rolls the frontier and checks event_id==position but performs no precursor_hash verification; crates/pardosa-nats/src carries zero precursor logic. Since gh-report runs the NATS backend in production (GH_REPORT_PARDOSA_BACKEND=nats), the deployed backend is the unverified one. Ratified remedy (roadmap adr-fmt-i8j15, amend PGN-0010 R8): lift the six on-read invariants into one backend-agnostic stage above the adapter seam so .pgno, NATS, and any future adapter inherit identical mandatory verification; apps only handle the typed error.
cherry-pit-merger is the command-side single-task serialiser, an alternative/complement to cherry-pit-app for command dispatch. ADR tree: CHE-0069 "cherry-pit-merger — Command-Side Single-Task Serialiser" [B] Accepted (References CHE-0017, CHE-0029, CHE-0030, CHE-0024, CHE-0042, CHE-0073), CHE-0085 "Merger/App Command-Side Choice Model" — no README found for the crate itself (gap noted in the prose digest). crates/cherry-pit-merger/Cargo.toml [dependencies] → cherry-pit-core, tokio, tracing — but no in-scope crate depends on cherry-pit-merger (not wired into cherry-pit-app's deps, nor gh-report's, nor any other in-scope Cargo.toml). Confirmed graph-orphan at C4 L2 discrepancies. 2026-07-20 correction: the orphan is anticipated by-design, not oversight — CHE-0085:R3 states "an absent family consumer of merger is expected state rather than architecture drift". Merger and App are siblings selected per aggregate concurrency profile (CHE-0085:R1/R2), so the missing edge is the ratified steady state until a consumer with a single-flight create-or-append profile wires it.
gh-report's runtime topology conforms to cherry-pit-app::App::run, with RunService/RepoService/WebhookService ApplicationServices, publish_or_trace, and a DomainEvent-carrying InProcessEventBus. gh-report DESIGN.md §4 (BC-9; CHE-0051) — but DESIGN.md's own corrigenda (W1/W2 truth-alignment notes) state several of these components "never shipped in this form": a self-documented intended-vs-actual gap in the prose itself, prior to any code check. Code-derived C4 L3 lists gh-report's actual app/ module set as state, daemon, collect, evidence_service, github_infra, team_refresh, webhook_context, work_queue, worker_pool, write_policy (crates/gh-report/src/app/*.rs) — the named ApplicationServices and publish_or_trace abstractions from DESIGN.md §4 do not appear at all, and the designed DomainEvent-carrying InProcessEventBus is likewise absent: the cherry_pit_app::InProcessEventBus type is wired (collect.rs:27,588–642) but only as InProcessEventBus::<SweepTimeoutEvent>, a scheduler sweep-timeout concern, never a DomainEvent bus — exactly the intended-vs-actual gap DESIGN.md §4's own W2 corrigendum records.

The prior pass's "CHE-0060 staged rollout stub pending PGN-0005 / removed when PAR-0021 lands" rationale is retracted as factually wrong: (1) the pardosa substrate already ships the precursor-hash chain — PGN-0005 is Accepted and implemented (integrity.rs:85 verify_precursor_chains, checked.rs:148, pardosa-wire/precursor.rs) with passing tamper-detection tests, not "pending"; (2) the cherry-pit capability port HashChainedEventStore (store.rs:298, exported lib.rs:75) is orphaned — zero impls, CHE-0060 Deprecated (stale 2026-05-19, host crate removed), PAR-0021 also Deprecated and retired without a PGN edge, and the PardosaEventStore the doc-comment names was retired whole (CHE-0073:R6 / CHE-0074:R8); (3) gh-report's live boot open (lifecycle.rs:863 open_with_backend → rehydrate_unchecked) does not reject discontinuity before replay, so SEC-0011:R3 is unmet at boot — precursor verification happens only on cursor reads (cursor.rs:265 stream_checked). The direction is abandoned/superseded, not pending — documented but the code diverges.
    2026-07-19 update — two corrections + a new backend-parity finding. (1) The "boot-load verification" candidate follow-up is itself mis-framed: SEC-0011:R3 mandates verification on-read, not a boot whole-stream pass; a literal boot verify exceeds R3 and risks a PGN-0005:R6 anchor-seam smell. (2) The narrower true defect: no precursor-hash-verifying open/rehydrate constructor is wired into gh-report's boot path — stream_checked (persist/checked.rs:208) is the sole precursor-hash site; it is reachable via open_validated (lifecycle.rs:1601 → rehydrate_validated → stream_validated, validated.rs:87 → stream_checked), so open_validated does run the chain check — but gh-report's live open never calls it (routing through open_with_backend/rehydrate_unchecked). (3) New, load-bearing: the precursor-hash chain is .pgno-only — the NATS/JetStream rehydrate arm (rehydrate_event_frames, lifecycle.rs:608) verifies contiguity + frontier but not precursor hashes, and pardosa-nats/src holds zero precursor logic, so the production backend (GH_REPORT_PARDOSA_BACKEND=nats) is the unverified one. The user has since ratified a mandatory-verification stance: no unchecked public read path on any backend; apps handle only the typed error. Roadmap adr-fmt-i8j15 (amend PGN-0010 with a new rule R8) lifts the six on-read invariants into one backend-agnostic stage above the storage-adapter seam, so .pgno, NATS, and a future Kafka adapter inherit identical behavior and the divergence closes by construction.

  • Framed against the Solon thesis: the repo's bet is that a small set of enabling constraints, enforced where the type system and build graph can enforce them, collapses the space of possible mistakes. Rows 11 and 13 are counter-evidence: places where intent outran shipped code, or wiring diverged from the stated rule. None of this contradicts the thesis — a subtractive architecture still needs its unshipped or partially-wired components named honestly, which DESIGN.md's own W1/W2 notes already do for row 13, and which the corrected N2 entry now does for the hash-chain capability.
