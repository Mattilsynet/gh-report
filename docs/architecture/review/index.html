Headline findings

  • The three spot-check invariants hold as MATCH, code-confirmed: sync public facade (row 1), the cherry-pit ⊥ pardosa sever (row 2), and the single-writer OCC fence (row 3) all show intended-vs-actual agreement with a precise code citation. This is the strongest Solon "illegal-is-uncompilable" evidence in the corpus: none of these three invariants rely on developer discipline alone — they are structurally enforced (sync signatures, empty dependency edges, a typed error variant on sequence conflict).
  • pardosa-nats ring purity (row 4) also holds exactly as specified — the dependency list in Cargo.toml matches the prose enumeration dependency-for-dependency, including the optional-feature carve-out for tempfile.
  • #[non_exhaustive] (row 5) holds where checked, but cherry-pit-gateway is an open gap, not a violation — the C4 survey grepped lib.rs rather than event_store/mod.rs, where the gateway's error type most likely lives. This is a survey-scope gap, distinct from the other DRIFT rows which reflect genuine documented-vs-shipped divergence.
  • gh-report as sole cross-substrate consumer (row 6) is UNDOCUMENTED — real and load-bearing in code (it is the only crate depending on both substrates) but never asserted as an explicit architectural claim in the prose sources surveyed. A DESIGN.md or ADR update naming this as an intended invariant would close the gap.
  • cherry-pit-merger is the sharpest divergence (row 7): two ADRs (CHE-0069, CHE-0085) describe a designed command-side serialiser role, but no in-scope crate wires it in. Whether this is dead/staged code or wired via a mechanism outside Cargo.toml (dynamic dispatch, feature flag, future integration) cannot be resolved from the dependency graph alone.
  • Deployment topology (row 8) is asserted operationally but absent from the repo — no Dockerfile, no Cloud Run manifest. This is squarely ASPIRATIONAL: intended in ops prose, not yet code-observable.
  • DESIGN.md's own W1/W2 corrigenda (row 9) are the strongest evidence the prose sources are self-aware of drift — the document names its own gap (ApplicationServices/publish_or_trace/InProcessEventBus never shipped) before any code comparison occurs. The C4 module inventory for gh-report's app/ confirms the admission: none of those names appear.
  • cherry-pit-sd-viz (row 10) is a clean MATCH for a deliberately-isolated tooling crate — its "standalone, zero pardosa dependency" self-description matches its actual graph isolation exactly.
  • Sibling-adapter symmetry (row 11) holds: cherry-pit-web and cherry-pit-projection both depend on core alone, as designed.
  • "cherry-pit-app depends on every other crate" (row 12) is narrower in practice — cherry-pit-merger and cherry-pit-storage sit outside that edge set. Combined with row 7, this suggests cherry-pit-merger's integration story is the single largest unresolved architectural question surfaced by this review.
  • Framed against the Solon thesis: the repo's bet is that a small set of enabling constraints, enforced where the type system and build graph can enforce them, collapses the space of possible mistakes. Rows 1-4 and 10-11 are exactly that promise redeemed — compile-time/dependency-graph facts that agree with the documented invariant, no runtime discipline required. Rows 7, 8, 9, and 12 are the counter-evidence: places where intent outran shipped code, or where wiring diverged from the stated rule. Neither set contradicts the thesis — a subtractive architecture still needs its unshipped or partially-wired components named honestly, which DESIGN.md's own W1/W2 notes already do for row 9.
  • Net verdict distribution (12 rows): 6 MATCH, 4 DRIFT, 1 UNDOCUMENTED, 1 ASPIRATIONAL — the review corpus is majority-confirming, with the DRIFT/gap rows concentrated in exactly the areas the source beads already flagged as gaps (cherry-pit-gateway non_exhaustive location, cherry-pit-merger wiring, DESIGN.md's self-admitted W1/W2 corrigenda, deployment topology).
