Design — Intended Architecture

Phase A: top-down, synthesized strictly from prose — ADRs, READMEs, DESIGN.md/OPERATIONS.md, AGENTS.md. No .rs source, no Cargo.toml, no dependency graph was read to produce this page. Sources: bd-adr-fmt-kgfyz (prose digest), adr-fmt-16ja3 (oracle binding-ADR summary).

Solon stance — why this repo exists

The repo styles itself "Solon": a small set of ratified, enabling constraints for agent-first development, after Solon the lawgiver. (AGENTS.md §Intent)

The bet is subtractive: "remove enough degrees of freedom that the remaining moves are obviously correct. When the type system rejects illegal architectures, the search space an agent must explore collapses." (AGENTS.md §Intent)

"The libraries are the product; the binaries are evidence they work." cherry-pit-* is the EDA+DDD+hexagonal substrate, engineered so illegal compositions — multiple writers per aggregate, async in the domain, leaky identity — do not type-check. pardosa* is the durable event-store substrate. adr-fmt/adr-srv are the governance plane (out of scope here). gh-report is the first non-trivial consumer: load-bearing proof the substrate carries real work. (AGENTS.md §Intent)

Niche (where to play): high-complexity, durable, intra-org workloads that do not need arbitrary horizontal scale-out. Single writer per aggregate; sync domain, async edges; linearizable per aggregate; sub-PB, single-region; crash-fail with recovery. "Webscale is a different game and is not played here." (AGENTS.md §Intent)

Non-goals (recorded so they aren't rediscovered): multi-region/multi-tenant webscale; frameworks/scaffolding/starter-kits (the constraints are the guidance); runtime-pluggable architectures — composition is compile-time CHE-0005:R1; schema evolution as a runtime feature; public-API surface freeze before v0.3. (AGENTS.md §Intent)

Per-family intended decomposition

gh-report family

gh-report — "GitHub organisation-evidence collector and HTML reporter built on the cherry-pit event-sourcing substrate." Alpha status. Persists to a native pardosa event store (crate::store::NativeStore) over pardosa/pardosa-schema/pardosa-nats. Binds a curated subset of cherry-pit's public surface per CHE-0073/CHE-0074: cherry-pit-wq (work-queue, worker-pool, budget, rate-limit), cherry-pit-storage (atomic writes, RunLock, snapshot signatures), cherry-pit-core (correlation context), cherry-pit-web (projection/serve), plus cherry-pit-gateway and cherry-pit-app (event-store adapter and root composition). Default backend: embedded .pgno; NATS/JetStream selectable.

  • Intended domain model CHE-0054 (superseded by CHE-0073, retained here as the as-designed reference): three DDD aggregates — Run (sweep lifecycle, CHE-0054:R1), Repo (repository evaluation lifecycle, CHE-0054:R2), WebhookDelivery (write-once, CHE-0054:R3). CHE-0073's storage remodel replaced the three-aggregate durable model with a single Repo logical stream, so the triad describes the original decomposition, not the shipped store layout. Read side: one CHE-0048 projection, EvidenceProjection, sole writer+reader of the read-model (CHE-0048:R2, CHE-0073:R7). (gh-report DESIGN.md §3; CHE-0054 stale, CHE-0073 live)
  • Runtime topology conforms to cherry-pit-app::App::run (gh-report DESIGN.md §4; BC-9; CHE-0051) — flagged divergence: DESIGN.md's own corrigenda (W1/W2 truth-alignment notes) state several of these components (RunService/RepoService/WebhookService ApplicationServices, publish_or_trace, a DomainEvent-carrying InProcessEventBus) "never shipped in this form" — an intended-vs-actual divergence already self-documented in the prose. This is exactly the class of gap the later Review phase is built to surface systematically.
  • Storage layout (gh-report DESIGN.md §5, OPERATIONS.md): <store_dir>/events/events.pgno (NativeStore, one fiber per repo domain key, CHE-0074:R4), org-events.pgno, team-events.pgno, locks/ (process fencing, CHE-0043). Production persistence: NATS JetStream; local default: .pgno.
  • Server: HTTP served via cherry-pit-web::serve generic pipeline; gh-report implements ServerState via AppState; extra route /api/v1/status.
  • Scoring contract (OPERATIONS.md): Repo Posture (arithmetic mean), Team Health (geometric mean, owner-level), Org Governance (geometric mean, org-level) computed from five shared controls (security policy, secret scanning, dependabot, branch protection, codeowners) plus lifecycle-derived rates (freshness, alert-free, archival coverage). Excluded status removed from the denominator, not counted as a fail.
  • Auth model: GitHub App > GITHUB_TOKEN > gh auth token fallback (local dev only). No offline/fixture mode — always reaches the GitHub API.
  • Governing ADRs: CHE-0073 (storage remodel), CHE-0074 (native pardosa store port), CHE-0022 (schema evolution), CHE-0039 (correlation propagation), CHE-0030 (flat public API), CHE-0054 (three-aggregate decomposition, superseded by CHE-0073), CHE-0048 (projection design), CHE-0051 (cherry-pit-app design), CHE-0018 (sync domain/async infra), CHE-0072 (storage-backend selector), CHE-0068 (projection freshness/coalescing), CHE-0092 (fold-side freshness), CHE-0088 (closed write-policy dispatch), CHE-0086 (generic read-serve), CHE-0062 (LayerLimits).

pardosa family (durable event-store substrate)

  • pardosa — "EDA storage layer implementing fiber semantics... runtime ring of the workspace — composes the substrate crates (pardosa-wire, pardosa-schema, pardosa-file) into a single adopter-facing appliance, pardosa::store::EventStore." Fiber = unit of identity (a strand of related events through which causality flows, PGN-0002). Event carries FiberId + EventId, both monotonic within scope; Dragline owns a dragline-local next_fiber_id allocator. Sole-interface seal per PGN-0008. StoreReader is !Send, StoreWriter is Send (PGN-0007). Governing ADRs: PGN-0002, PGN-0008, PGN-0004, PGN-0003.
  • pardosa-derive — proc-macro ring. #[derive(GenomeSafe)] walks struct/enum shape, emits pardosa_wire::sealed::Sealed + pardosa_schema::EventSafe + pardosa_schema::GenomeSafe (SCHEMA_HASH: u128) + Encode/Decode impls. Rejects serde attributes at type/variant/field level once derived; enums must be #[repr(u8)] with explicit discriminants; unions rejected (EVT-001). Governing ADR: PGN-0003.
  • pardosa-fiber-store — no dedicated README located within scope; referenced by gh-report DESIGN.md as pardosa_fiber_store::FiberStore<DomainEvent>, the native EventStore adapter gh-report wires. Extraction ruled by CHE-0084:R7. gap
  • pardosa-file — "Payload-opaque .pgno file container: writer, reader, format constants, and the Syncable durability seam." On-disk substrate ring; defines the byte-level .pgno format; verified on Reader::open (magic, FORMAT_VERSION, footer checksum, ascending offsets, per-message xxh64 checksums). Payload-opaque: carries schema_hash: u128 in the header but never names the payload type (typing lives one ring up, in pardosa-schema/pardosa::store). Three writer flavours: Writer, AppendWriter, PageClass. "Drop is not a durability boundary" — callers must fence via sync_data. Governing ADRs: PGN-0004, PGN-0003.
  • pardosa-nats — no dedicated README found (gap). The ring-purity invariant (only tokio/async-nats/bytes/blake3/futures-util deps) is the primary intended-architecture claim available in prose; see Invariant 3 below. gap
  • pardosa-read — "Read-only RON reader for pardosa events stored in JetStream. Built for agent-driven debugging against production NATS." Structural read-only guarantee: constructs only JetStreamHandle::replay_readonly reads; never calls append/sync/get_or_create_stream/update_stream; a missing stream surfaces as a typed error rather than being provisioned (PGN-0008:R8 rehydrate-only-errors-on-never-created). CLI tool, not library-consumed by other in-scope crates per this prose survey.
  • pardosa-schema — "Typed-payload vocabulary for pardosa events: GenomeSafe marker, bounded string/byte/vec wrappers, total-order float wrappers, CharScalar." Sits one ring above pardosa-wire: builds payload-typed vocabulary atop wire's payload-opaque codec traits. Every type is Send+Sync (asserted by test). Bounded wrappers (EventString etc.) — the MAX const is part of schema identity; changing it is a schema migration (PGN-0003 + PGN-0009). The derive feature re-exports pardosa_derive::GenomeSafe. Governing ADRs: PGN-0003, PGN-0009.
  • pardosa-wire — "In-house canonical encoding format for pardosa events (PGN-0003)... substrate-purity layer: a #![no_std] + extern crate alloc codec." Encode/Decode traits + to_vec/from_bytes/ from_bytes_with_cap helpers; bounded decode (DEFAULT_DECODE_CAP) so adversarial input can't force unbounded allocation ("PGN-0003 cap discipline"). EventSafe marker + a sealed module gate participation; downstream crates (pardosa-schema, pardosa-derive) extend the sealed graph rather than opening it. Dependency-free by default; uuid/blake3 behind features.
  • pardosa-mint-private-tests test-support — no README found within scope glob; not documented in prose sources searched. gap
  • pardosa-test-support-harness test-support — no README found within scope glob. gap

cherry-pit family (EDA+DDD+hexagonal substrate)

  • cherry-pit-core — "Foundational traits for cherry-pit: aggregates, commands, events, ports." Every infra port (EventStore, EventBus, CommandBus, CommandGateway) is bound to a single aggregate type via associated types — "the compiler enforces end-to-end type safety from command dispatch through event persistence and publication." Domain traits are sync (Aggregate, HandleCommand, DomainEvent, Command, Policy, Projection); port traits are async RPITIT, no dynamic dispatch. Dependencies: only serde, uuid, jiff — no async runtime, no framework deps. Status: implemented, stable for v0.1.
  • cherry-pit-app — "Root composition crate wiring Aggregate/Policy/Projection against EventStore/EventBus/CommandGateway." The only crate sanctioned to depend on every other cherry-pit crate (CHE-0029 acyclic-DAG rule). Flat public API via crate-root pub use (CHE-0030:R1). Central type: App<G,S,B,P,D> (five type params, no Box over infra ports, CHE-0005:R1). Dead-letter contract: terminal-category policy errors → DeadLetterRecord → DeadLetterSink::record; the publish loop does not abort on one bad envelope (CHE-0051:R7 + CHE-0046:R2). One App per aggregate (CHE-0051:R9). App::run requires a multi-thread tokio runtime (the sync bus callback bridges via Handle::block_on). Governing ADRs: CHE-0005, CHE-0017, CHE-0024, CHE-0030, CHE-0038, CHE-0039, CHE-0040, CHE-0046, CHE-0048, CHE-0051 (primary design ADR).
  • cherry-pit-gateway — "Infrastructure implementations for cherry-pit: event stores." Provides MsgpackFileStore — a file-based MessagePack event store with atomic writes (CHE-0032), process-level fencing (CHE-0043), optimistic concurrency (CHE-0006, CHE-0035). One .msgpack file per aggregate, rewritten on every append (CHE-0036:R1/R2). Advisory file lock for single-process ownership. RUNBOOKS.md documents operational recovery: R1 orphan temp-file recovery, R2 corrupt-data classification, R3 quarantine-before-repair, R4 dead-letter schema, R5 stale-lock recovery, R6 migration recovery (CHE-0047). Status v0.1.0: sole event-store implementation; object_store-backed stores planned (CHE-0044).
  • cherry-pit-merger — no README found within scope (gap). ADR tree shows CHE-0069 "cherry-pit-merger — Command-Side Single-Task Serialiser" [B] Accepted (references CHE-0017, CHE-0029, CHE-0030, CHE-0024, CHE-0042, CHE-0073) and CHE-0085 "Merger/App Command-Side Choice Model" — intended role per ADR title: a command-side single-task serialiser, an alternative/complement to cherry-pit-app for command dispatch. gap
  • cherry-pit-projection — "Projection drivers and storage backends for cherry-pit read models... realises CHE-0048." Read-side adapter layer: folds durable event streams into query-optimized projection state, persists checkpointed snapshots for restart/rebuild. Sibling adapter crate to cherry-pit-gateway; depends on cherry-pit-core only, does not stack on the gateway implementation. Key types: ProjectionDriver<P,S>, InMemoryProjection, FileProjectionStore (snapshot-then-checkpoint write order — a crash between the two means "rebuild", not "trust snapshot", CHE-0048:R2), ProjectionCheckpoint, ProjectionError (#[non_exhaustive]).
  • cherry-pit-storage — "Synchronous filesystem primitives — atomic writes, run-locks, content-addressable signatures — for cherry-pit consumers." Per CHE-0053 absorbs error/fs/lock/signature modules from the donor quics-memoization crate. Flat public API over private modules (CHE-0053:R3). Status: implemented v0.1.
  • cherry-pit-web — "HTTP adapter exposing the cherry-pit CommandGateway over axum... realises CHE-0049 (design) ... and CHE-0050 (CommandRouter port)." Role: HTTP-edge adapter for command-side traffic — owns status mapping, correlation echo, idempotency-key threading, versioned /v1/ DTO contract; explicitly does not own auth, WebSocket surfaces (default build), or static-content caching (consumer-attached via extra_routes). Sibling adapter crate to cherry-pit-projection, depends on cherry-pit-core only. AppState<G,S,R> is generic typed state, no Box. Optional projection feature adds a second read-side router (build_projection_router) — HTTP snapshot fetch + narrowed WS push for deltas, structurally sealed against dyn use (ProjectionSource, CHE-0049:R12 + CHE-0005:R1). Out of scope v0.1: auth, transports beyond HTTP/1.1+JSON on the command surface (WS only on the projection feature), static content caching, idempotency persistence.
  • cherry-pit-wq — "Domain-agnostic concurrency and resource-pacing primitives for cherry-pit consumers." Per CHE-0055 (supersedes CHE-0052) absorbs work_queue, worker_pool, budget, rate_limit from the donor quics-aggregate crate; pagination explicitly not in this crate (relocated to gh-report::github::pagination, CHE-0055:R7). Status implemented v0.1, flat re-export surface (CHE-0055:R10).
  • cherry-pit-sd-viz tooling — "standalone, client-side, animated discrete-event simulation of gh-report's runtime queue network." No dependency on a running gh-report; no trace capture; wasm/browser-only. Documents the intended gh-report runtime topology narratively: three triggers (scheduled sweep via SweepSaga/SweepPhase; webhook via webhook_handler enqueuing JobSource::External; warm start via warm_start_from_baseline bypassing the queue) feeding a per-packet WRITE side (WorkQueue → worker_loop/ LiveEvaluator::evaluate → JobOutcome → delivery_loop → record_repo folds EvidenceProjectionEvent), joined by a BARRIER (BatchTracker — a sweep run blocks until all ScheduledBatch jobs of that run drain; webhook jobs never gate), then a per-RUN READ side (finalize_and_publish → build_cached_pages → commit_cached_pages via atomic ArcSwap swap+generation++ → PageUpdateEvent broadcast once per run finalize), plus a continuous SERVE path (cache_fallback reads the current ArcSwap generation independent of run completion). Strong prose-derived architecture claim for gh-report's dataflow.

Cross-family intended dependency boundaries

  • gh-report depends on the curated cherry-pit subset per CHE-0073/CHE-0074 — cherry-pit-wq, cherry-pit-storage, cherry-pit-gateway, cherry-pit-app, cherry-pit-core, cherry-pit-web — plus the native-store pardosa set: pardosa, pardosa-fiber-store, pardosa-file, pardosa-schema, pardosa-nats. (gh-report README "Status" section; full edge list cross-checked against the C4 L2 dependency table)
  • cherry-pit-app is the only cherry-pit crate sanctioned to depend on every other cherry-pit crate (CHE-0029 acyclic-DAG rule). (cherry-pit-app README)
  • cherry-pit-web and cherry-pit-projection are sibling adapter crates, both depending only on cherry-pit-core, neither stacking on cherry-pit-gateway. (cherry-pit-projection README, cherry-pit-web README)
  • cherry-pit does not depend on pardosa — gh-report is the point where both substrates are consumed together, not via a cherry-pit → pardosa edge. See Invariant 4 below for the precise enforcement chain.
  • pardosa-wire (no_std, dependency-free core codec) < pardosa-schema (adds payload-typed vocabulary) < pardosa (adopter-facing appliance composing pardosa-wire + pardosa-schema + pardosa-file) — ring ordering per pardosa-schema/pardosa-wire READMEs ("sits one ring above").
  • pardosa-derive is a proc-macro ring parallel to/consumed-by pardosa-schema (extends the sealed graph pardosa-wire opens).
  • pardosa-nats is architecturally isolated (ring-purity invariant) from cherry-pit-* and the pardosa adapter ring — see Invariant 3.

Intended Invariants

The ten invariants below are the load-bearing set a later Review phase verifies against code. Each cites the exact ADR id(s) confirmed verbatim via adr-fmt --context/--refs (adr-fmt-16ja3).

1. Synchronous public facade

No async fn on the public surface of pardosa::store/prelude (PGN-0010:R5, PGN-0008, PGN-0015:R6). Rationale: backends own async internals; adopters get a synchronous appliance. The sync-over-async bridge is pardosa-nats/src/handle.rs::run_op (block_on); std::sync::Mutex behind the facade is deliberate — not to be "fixed" to tokio::sync::Mutex (would break single-writer linearizability).

1b. Store module = sole public surface

pardosa::store (+ prelude) is the sole-interface seal (PGN-0008:R1, PGN-0004): adopters reach the journal exclusively through pardosa::store::EventStore and pardosa::prelude. Internal substrate types are never named in adopter code; drift is pinned by tests/ui_pass/prelude_usable.rs.

2. run_op block_on bridge

PGN-0010:R5 — the sync-over-async bridge is a single named mechanism (run_op), not an ad-hoc pattern scattered across call sites. Rationale: one audited crossing point between the sync facade and the async NATS backend keeps the bridge's correctness provable in one place.

3. Substrate ring purity

PGN-0001:R2 — pardosa-nats depends only on tokio, async-nats, bytes, blake3, futures-util (+ tempfile behind an optional test feature): no cherry-pit-* or pardosa adapter-ring edges; instrumentation/metrics belong in the pardosa adapter ring, never in pardosa-nats. High-cardinality ids (event_id, ack, stream) go on spans/logs, never metric labels (COM-0019:R6). Rationale: substrate crates that stay dependency-minimal are easier to audit for the compile-time guarantees the rest of the design leans on.

4. cherry-pit ⊥ pardosa sever

cherry-pit does not depend on pardosa. Naming note (per the oracle summary): "CHE-0010 sever" as used in casual conversation about this repo is imprecise — CHE-0010 ("DomainEvent Supertrait Bounds") is the contract governing what bounds a DomainEvent must satisfy (Clone+Send+Sync+'static+Serialize+DeserializeOwned, CHE-0010:R1..R3), not the non-dependency rule itself. The actual cherry-pit → pardosa non-dependency is enforced by CHE-0029:R4/R6 (core deps limited to serde/uuid/jiff; CI cargo-tree purity assertion), CHE-0084:R5, and CHE-0086:R7, plus the async-trait/pardosa CI tripwires (see also AGENTS.md's async-trait-tripwire and gh-report-projection-lock-tripwire jobs). Rationale: the substrates model genuinely separate concerns (domain events vs. durable storage bytes); gh-report is the single point where both are consumed together, never via an inter-substrate edge.

5. Single-writer-per-aggregate/stream

"Single-writer-per-stream eliminates concurrent mutation at the architectural level, not through runtime locks" (SEC-0006:R2, foundation-level, applies to all crates). COM-0018 "Single-Writer Concurrency as Foundational Invariant" references PGN-0010, CHE-0006. CHE-0006 "Single-Writer Assumption Per Aggregate." PGN-0016:R1/R2/R7 — JetStream subject-sequence OCC fence: detection-not-prevention, abort+replay, no in-band retry. Rationale: consistency is achieved via the fence, not consensus — cheaper and matches the sub-PB single-region niche.

6. #[non_exhaustive] on error/domain enums

Mandated on error enums (PGN-0006:R1..R7, CHE-0021 scoped to error types, RST-0006:R1) — adding variants is non-breaking, removing them is; not for serde DTOs. Sharpened by the oracle summary: the rule inverts for cherry-pit domain event enums — CHE-0022:R5 requires domain event enums are not #[non_exhaustive] (exhaustive match is forced so a new event variant cannot be silently unhandled), and gh-report's WritePolicyCategory dispatch enum is likewise deliberately closed (CHE-0088:R1, no catch-all arm). Rationale: errors grow forward-compatibly; domain events and closed dispatch enums must be exhaustively handled so new variants can't be silently dropped.

7. RPITIT-only / no async-trait

All async port traits use return-position impl Trait in traits (RPITIT), never async-trait, never Box<dyn Trait> (CHE-0025:R1/R2, enforced by the async-trait-tripwire CI job). Applies to EventStore/EventBus/CommandBus/CommandGateway (CHE-0070/CHE-0076 listable/history extensions likewise forbid trait objects) and to cherry-pit-web's read-serve adapters (CHE-0086:R6). Rationale: static dispatch keeps the compile-time type-safety chain (CHE-0002 "Make Illegal States Unrepresentable") intact end-to-end; a boxed trait object would reintroduce a runtime escape hatch the design is built to close.

8. gh-report projection-lock discipline

Checkpoint written strictly after snapshot (CHE-0048:R2) — a crash between the two means "rebuild", never "trust snapshot." The lock chokepoint is AppState::lock_projection(); raw .projection_state.lock( calls outside crates/gh-report/src/app/state.rs are banned and enforced by the gh-report-projection-lock-tripwire CI job. Rationale: a single audited lock acquisition point makes the single-writer discipline on the read-model provable by grep, not just by convention.

9. Atomic-write protocol

CHE-0032:R1..R4 — temp file → File::sync_all → rename → parent-directory fsync. Applies wherever cherry-pit-storage or cherry-pit-gateway persist to the filesystem (gateway's MsgpackFileStore, gh-report's RunLock/snapshot signatures). Rationale: survives a crash between write and rename without leaving a torn or partially-visible file; matches the "crash-fail with recovery" niche stance.

10. Cross-cutting foundation invariants

Binding on all three families, not family-specific: GND-0010:R1..R7 (PC/EC PACELC — consistency via single-writer per COM-0018, not consensus; OCC detection; sync domain / async edges only; silent consistency violation must surface); GND-0011:R1..R7 (read-model bounded-stale/monotonic/causal-floor/read-your-writes; no derived view may author truth — that would be a second writer, defeating the fence); RST-0001..0006 (pinned toolchain/MSRV, workspace lints as the pedantic standing bar, workspace deps + cargo audit/deny, #![forbid(unsafe_code)] every crate, the non-exhaustive-check tool as sole enforcement for RST-0006:R1).

Self-documented intended-vs-shipped gaps

gh-report's own DESIGN.md carries W1/W2 truth-alignment corrigenda stating that several designed components — the RunService/RepoService/WebhookService ApplicationServices, the publish_or_trace function, and a DomainEvent-carrying InProcessEventBus — "never shipped in this form." These are recorded here verbatim because they are exactly the class of divergence a systematic Review pass (intended design vs. independently-derived code survey) is designed to surface across all three families rather than only where an author happened to leave a corrigendum. Any reader of this Design doc should treat the runtime-topology description above (§gh-report family) as as-designed narrative, not a claim about what ships today.
