Cross-substrate dependency

gh-report is the sole cross-substrate consumer — it depends on BOTH pardosa and cherry-pit (crates/gh-report/Cargo.toml [dependencies]). The two substrates do not depend on each other in either direction: zero "pardosa" dependency lines across all 9 cherry-pit-*/Cargo.toml files (only a code comment in cherry-pit-core's noting pardosa-encoding was removed, and a self-description string in cherry-pit-sd-viz's), and zero "cherry" matches across all pardosa*/Cargo.toml files.

L1 discrepancies

  • Deployment topology lives out-of-tree: not observable from the gh-report repo alone (no Dockerfile at root), but fully declared in the companion gh-report-ops repo — Cloud Run + IAP + NATS JetStream via applied Terraform (gh-report-ops/cloudrun/ghreport/main.tf). A code-only survey scoped to the single repo mis-reads this as a gap; the C4 boundary for a deployed service must include its sibling ops repo.
  • cherry-pit-sd-viz is dependency-graph-isolated: its relationship to the rest of the cherry-pit family is asserted only by its own Cargo.toml description string, not by any actual dependency edge — no in-scope crate depends on it, and it depends on none in-scope observed.
  • cherry-pit-merger orphan propagates to L1: per the L2 discrepancy, unclear whether cherry-pit-merger is an active member of the cherry-pit context or dead/staged code from code evidence alone.
